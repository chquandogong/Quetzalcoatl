# GarryTan Office Hours Result — v1.2.0 업데이트

> 상태: 확정 · 날짜: 2026-06-23 · 소유자: GT Office Hours Partner · 승인: CHENGHAO QUAN

## 원래 요청

스킬에 5개 업데이트가 의미 있는지부터 검토하고 작업하라: ① 묻지 말고 끝까지 ② 레이트리밋 대기·재개(하위 에이전트 포함) ③ 문서 체계화 ④ git 생성·커밋·푸시·태그 ⑤ live artifact 대시보드를 에이전트/기기 간 공유 표면으로.

## 재정의된 문제

5개의 개별 기능 요청이 아니라, **"한 세션·한 기기·한 번의 실행"이라는 경계를 넘는 능력**이 없다는 하나의 문제다. 진짜 고통: 긴 작업이 확인·한도·세션·기기 경계에서 끊기고, 그 상태를 넘길 방법이 없다.

## 진짜 사용자 / 강한 고통의 증거

- 스킬 소유자가 Claude Code에서 긴 자율 작업을 돌릴 때: 매 단계 확인으로 끊김, 한도로 유실, 데스크탑→모바일 인계 불가.
- 이번 요청 자체가 증거다("묻지 말고 끝까지", "기다렸다 계속", "이어서·나눠서").

## AI를 쓸 이유

- 자율 실행·재개·문서화·핸드오프 조율은 사람이 수동으로 하면 느리고 누락된다.
- AI가 틀릴 때 피해 → **안전 게이트(§1.6)와 커밋 이력**으로 제한.

## 만들지 말아야 할 것 (지금)

- 진짜 락 매니저/분산 합의(프롬프트로 불가, 과대설계).
- 게이트를 없애는 "완전 자동". 자율 ≠ 게이트 해제.
- 강제되지 않는데 "보장된다"고 말하는 기능.

## 가장 작은 쐐기

SKILL.md에 5개 규율을 **상호 모순 없이** 추가하고, 가장 위험한 자율 실행 문구만 독립 에이전트로 RED/GREEN 검증한다. 나머지는 정직한 규율+환경별 강등으로 표현.

## 3~5개 실행 대안 → [ALTERNATIVES](../02-decisions/ALTERNATIVES.md)

요지: 자율 모델은 "스텝 단위 자율"이 아니라 **"승인된 계획 범위 내 자율 + 허용목록"**으로, 핸드오프는 "락 없는 멀티라이터"가 아니라 **"단일 라이터 / commit-as-CAS"**로.

## 추천 방향

5개 전부 채택하되 위 두 재정의를 적용. 모델 중립 강등을 각 기능에 복제.
