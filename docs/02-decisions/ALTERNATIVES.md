# DECISION OPTIONS — 핵심 두 결정의 대안

## 결정 1 — 자율 실행 모델 (관련: D1, D4)

선택 기준: 안전 · 정체성 정합 · 요청 충족 · 단순성 · 강제 가능성.

| 대안                             | 설명                                | 장점                  | 단점/실패 조건                            |
| -------------------------------- | ----------------------------------- | --------------------- | ----------------------------------------- |
| A. 스텝 단위 무조건 자율         | 모든 가역 스텝 자율                 | 요청 ① 문자 그대로    | §1.1·§15 위반, 회색지대 폭주, 정체성 전복 |
| B. 승인된 계획 범위 내 자율 ✅   | 1회 의미 승인 후 범위 내 자율       | 의미먼저 양립·안전    | 위임 범위 해석 필요                       |
| C. 허용목록 기반 게이트 ✅(보강) | 경로/명령/행동 화이트리스트 밖 정지 | 회색지대 제거, 강제적 | 목록 작성 부담                            |
| D. 자율 도입 안 함               | 기존 매단계 확인 유지               | 가장 안전             | 요청 미충족, 긴 작업 끊김                 |

| 기준        |   A |   B |   C |   D |
| ----------- | --: | --: | --: | --: |
| 안전        |   2 |   4 |   5 |   5 |
| 정체성 정합 |   1 |   5 |   5 |   3 |
| 요청 충족   |   5 |   4 |   4 |   1 |
| 단순성      |   4 |   4 |   3 |   5 |
| 강제 가능성 |   2 |   3 |   4 |   5 |

**추천**: **B + C 결합** — 계획 승인으로 정체성 양립(B), 허용목록으로 회색지대 차단(C). A는 위험, D는 요청 미충족이라 버림. 남은 리스크: 위임 범위/허용목록 작성 → 사용자가 `/autonomy`로 조정.

## 결정 2 — 핸드오프 대시보드 동시성 (관련: D2)

| 대안                        | 설명                             | 장점             | 단점/실패 조건                         |
| --------------------------- | -------------------------------- | ---------------- | -------------------------------------- |
| A. 멀티라이터 자유쓰기      | 누구나 보드에 쓰기               | 단순·병렬        | lost update·머지충돌·조용한 손상       |
| B. 단일 라이터 ✅           | Orchestrator만 쓰기, 나머지 읽기 | 손상 원천 제거   | 병렬 쓰기 불가(제안으로 우회)          |
| C. commit-as-CAS ✅(필요시) | 커밋 안착으로 claim 확정         | 충돌 가시화·병렬 | push 경합 잔존(충돌로 표면화)          |
| D. 외부 DB/큐               | 진짜 동시성 제어 도입            | 정확한 락        | 프롬프트 범위 밖·모델 중립 위반·과설계 |

**추천**: 기본 **B**, 진짜 병렬이 필요하면 **C**. A는 데이터 손상, D는 과설계라 버림. SSOT=git repo, live artifact=읽기 미러. 남은 리스크: 동시 push 경합(드묾) → "먼저 안착 커밋이 이긴다".
