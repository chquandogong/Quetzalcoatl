# PROJECT REVIEW — Quetzalcoatl

> 상태: 검토 완료 · 날짜: 2026-06-29 · 소유자: Codex/Quetzalcoatl · 승인: 사람 결정 대기

## 한 줄 결론

**버리지 말고 취하되, 프로젝트를 "대형 범용 OS"로 더 키우는 것은 멈춘다.** 다음 사이클은 신규 기능이 아니라 **Core Contract 분리, 설치/문서 현재성, 평가 자동화**에만 써야 한다.

## 내가 이해한 프로젝트

Quetzalcoatl은 아이디어·개발·글쓰기·조사 작업을 바로 실행하지 않고, 의미 검증 → Office Hours → 타당성 → 대안 비교 → 교차검증 → 문서화 → Ship/Retro까지 묶는 모델 중립형 Agent Skill이다. 현재 저장소 기준 최신 소스는 `v1.6.0`이며, `/context` 벤더 중립 브리핑까지 포함한다.

## 근거

| 영역 | 판단 | 근거 |
| ---- | ---- | ---- |
| 문제 가치 | 높음 | 긴 agentic 작업에서 의미 검증, 안전 게이트, 재개, 핸드오프, 문서화가 빠지는 문제는 실제 반복 고통이다. |
| 시장/도구 적합성 | 높음 | Codex와 Claude Code 모두 skills/plugins, reusable workflows, project guidance, subagents 방향으로 가고 있다. |
| 구현 완성도 | 중간~높음 | 매니페스트/SKILL 버전 1.6.0 일치, JSON 유효, 섹션 §0~§24 연속, RUNBOOK/DASHBOARD/HTML 미러 1.6.0 동기화가 로컬 검증됨. |
| 사용성 | 중간 | `SKILL.md`가 약 1,490줄로 길다. 공식 Agent Skills 권장 구조상 full body가 활성화 시 전부 로드되므로 Core/부록 분리가 필요하다. |
| 운영 리스크 | 중간 | 저장소는 1.6.0이지만 Codex 개인 스킬 설치본은 1.4.2였다. docs/README, PROJECT_BRIEF, TEST_PLAN, RISK_REGISTER 일부는 v1.4/v1.5 상태로 남아 있다. |

## 버릴 것

| 대상 | 이유 |
| ---- | ---- |
| 더 많은 모드 추가 | 이미 충분히 넓다. 새 기능은 핵심 규칙 준수율을 떨어뜨릴 가능성이 크다. |
| "AI 팀 OS"라는 과대 범위 | 실제 강점은 모든 운영을 대신하는 것이 아니라, 어려운 프로젝트 시작과 안전한 실행 판단을 구조화하는 것이다. |
| 프롬프트만으로 강제 보장한다는 인상 | 스킬은 규율이며, 실제 강제는 환경 훅·CI·권한·테스트에 연결될 때만 가능하다. |
| stale 설치/문서 방치 | 실제 호출 경로가 오래되면 프로젝트 가치가 바로 사라진다. |

## 취할 것

| 대상 | 이유 |
| ---- | ---- |
| Meaning-first + Office Hours | 차별점이 가장 선명하다. "바로 만들기"를 막는 게 이 스킬의 존재 이유다. |
| 안전 게이트 + 허용목록 | 자율 실행과 인간 승인 경계를 설명하는 핵심 자산이다. |
| Decision Log / Dashboard / `/context` | 장기 작업과 핸드오프에 실질 가치가 있다. |
| 실제 교차검증 기록 | v1.5.0/v1.6.0에서 단일모델 사각을 잡은 증거가 있어 프로젝트의 방법론을 뒷받침한다. |

## 선택지

| 선택지 | 장점 | 단점 | 판정 |
| ---- | ---- | ---- | ---- |
| A. 중단/보관 | 유지비 0 | 이미 만든 자산과 실제 문제 해결력을 버림 | 비추천 |
| B. 개인용으로만 유지 | 가장 현실적, 과대마케팅 방지 | 외부 사용자 성장 제한 | 가능 |
| C. 공개 스킬로 유지하되 Core 분리 | 가치 유지 + 사용성 개선 | 한 번의 구조 정리 필요 | **추천** |
| D. 플랫폼/앱으로 확장 | 강제력·UX 개선 가능 | 지금은 검증 전 과투자 | 보류 |
| E. 여러 도구별 네이티브 버전으로 분기 | 각 환경 최적화 | 모델 중립성 붕괴, 유지비 증가 | 비추천 |

## 추천 방향

**C: 공개 스킬로 유지하되 Core Contract 분리.** 단기 목표는 기능 확장이 아니라 다음 3개다.

1. `SKILL.md`를 Core Contract(~250~400줄) + references로 분리한다.
2. Codex/Claude 설치본이 저장소 최신과 맞는지 검증하는 로컬 체크를 추가한다.
3. `evals/`를 만들어 trigger eval + 행동 eval 5~10개를 관리한다.

## 중단 조건

- Core 분리 후에도 실제 사용에서 과도한 프로세스 때문에 매번 우회한다.
- 설치본 stale이 반복되어 최신 스킬이 실제로 호출되지 않는다.
- eval에서 의미 검증/게이트/대안 비교가 없는 기본 모델 대비 유의미한 개선을 못 보인다.
- 외부 사용자가 "너무 길고 무겁다"는 피드백을 반복한다.
